## 四、代码评审标准

### 1. 评审时机
- 每个任务完成后、部署前必须评审
- 评审通过才能进入下一任务

### 2. 评审内容

#### 功能评审
- [ ] 功能符合需求
- [ ] 数据正确性
- [ ] 无功能遗漏

#### 代码质量
- [ ] 无TypeScript编译错误
- [ ] 无console.error
- [ ] 变量命名清晰
- [ ] 关键逻辑有注释

#### 性能与安全
- [ ] 无明显性能问题
- [ ] 无敏感信息泄露

#### 部署验证
- [ ] 本地build通过
- [ ] 页面可正常加载

### 3. 评审人
- 龙六的代码 → 我评审
- 扣钉的代码 → 龙六评审
- 测试结果 → 我验收

### 4. 评审结果格式
```
## 评审结果
### 通过/不通过
### 问题
1. xxx
### 修改建议
- xxx
```

## 五、主动推进机制

### 检查频率：每10分钟

### 检查内容
1. 所有任务状态
2. 阻塞任务
3. 完成进度

### 状态流转
```
pending → in_progress → reviewing → done
              ↓
         blocked
```

### 触发动作

| 状态 | 动作 |
|------|------|
| in_progress > 20分钟 | 询问进度 |
| blocked | 协调解决或重分配 |
| reviewing > 15分钟 | 催评审 |
| done | 确认并进入下一任务 |

### 汇报机制
- 每30分钟汇报整体进度
- 遇到阻塞立即汇报
- 里程碑完成立即汇报

## 六、风险预案

| 风险 | 应对 |
